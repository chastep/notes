# Sterling → BookMule Gap Analysis & Remaining Build Plan

## Summary

This session audited the full Sterling codebase against BookMule's feature set to identify what's wired end-to-end versus what's still missing. Import (CSV → QBO), mass-edit, mass-delete, and export are all fully wired with real Inngest background jobs and QBO API calls — significantly more complete than expected. The strategic context from Will's notes (target accountants not direct SMB; vendor canonicalization is the moat; Plaid bank feeds are V1 priority; IRS applications need to be filed now) was incorporated into the gap analysis. The plan was updated with a prioritized build list covering 17 remaining items across P0 (critical path), P1 (differentiators), P2 (feature completeness), and P3 (compliance/production).

## Key Topics

- Full codebase audit: pages, API routes, Inngest functions, Prisma models
- BookMule feature set: 8 core capabilities including safe merge, invoice templates, AP/Bill Pay, 1099 Center
- Will's strategic notes: accountant ICP, per-vendor pricing, vendor canonicalization engine as moat, Plaid as V1 must-have
- QBO token refresh gap — blocks all operations after ~1 hour of inactivity
- 1099 filing flow: W-9 chase emails, TIN validation, 1099-NEC PDF generation, IRS FIRE export
- Vendor canonicalization: fuzzy match + embeddings + LLM reasoning to unify "Joe's Plumbing LLC" / "JOSEPHS PLUM ACH" / "Joe Smith Plumbing"

## Decisions & Insights

- Import, mass-edit, mass-delete, and export are production-ready — the core QBO write operations are done
- Receipt scan and 1099 Center are "UI + data model only" — no real AI processing or filing wired
- Safe merge (consolidate vendors/customers/accounts) is completely absent from the codebase
- QBO token refresh is the single most urgent fix — every operation will fail after ~1 hour without it
- "Reverts are free" is a marketing claim with no backend implementation yet
- The strategic differentiation (vs Track1099, Tax1099 etc.) is vendor canonicalization + bank-feed cross-reference, not the filing itself
- IRS TIN Matching e-services and FIRE/IRIS Transmitter Control Code applications need to be filed immediately (4-8 week approval lag)
- Plaid integration is a V1 priority per Will — without bank feeds the product is "a glorified Track1099"

### QBO Token Refresh — helper to add to src/lib/providers/qbo.ts
```typescript
export async function ensureFreshToken(entity: Entity): Promise<Entity> {
  if (entity.tokenExpiresAt && entity.tokenExpiresAt > new Date()) return entity;
  const res = await fetch('https://oauth.platform.intuit.com/oauth2/v1/tokens/bearer', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded',
      Authorization: `Basic ${Buffer.from(`${process.env.QBO_CLIENT_ID}:${process.env.QBO_CLIENT_SECRET}`).toString('base64')}`,
    },
    body: new URLSearchParams({ grant_type: 'refresh_token', refresh_token: entity.refreshToken }),
  });
  const data = await res.json() as { access_token: string; refresh_token: string; expires_in: number };
  return db.entity.update({
    where: { id: entity.id },
    data: {
      accessToken: data.access_token,
      refreshToken: data.refresh_token,
      tokenExpiresAt: new Date(Date.now() + data.expires_in * 1000),
    },
  });
}
```

### What we worked on
This session started as a continuation from a prior context (QBO Bill import fixes + rerun button) and pivoted into a full product gap analysis. The user wanted to understand what's left to build to reach BookMule feature parity, informed by both the current codebase and Will's strategic notes (`notes/will-thoughts-052326.md`). We fetched BookMule's marketing site to enumerate their 8 core capabilities, did a full codebase audit (pages, APIs, Inngest functions, Prisma models), read the actual Inngest function files to confirm what's genuinely wired vs stub, and then cross-referenced against Will's strategic notes about the accountant ICP and vendor canonicalization moat. The gap analysis was written into the plan file at `/Users/chasestephens/.claude/plans/resilient-orbiting-scroll.md`.

### Key decisions and why
- **Token refresh is P0**: Every QBO operation will fail after ~1 hour without it. It's a silent failure mode that would block real usage.
- **Revert operations are P0**: The product's marketing says "reverts are free" and the history UI shows this promise — but no backend exists. This is a trust-breaking gap.
- **Vendor canonicalization is the moat** (per Will): The differentiation vs filing-only tools (Track1099, Tax1099) is resolving "Joe's Plumbing LLC" = "JOSEPHS PLUM ACH" = "Joe Smith Plumbing" across QBO + bank feeds. Without this, the product has no durable advantage.
- **Plaid is V1 not V2** (per Will): Bank feeds are required for the canonicalization engine to cross-reference. Without bank data, the product can't do the cleanup work that justifies the pricing premium.
- **IRS applications need to be filed this week** (per Will): TIN Matching e-services and FIRE/IRIS TCC both have 4-8 week approval lags. Filing late means V1 ships with format-only validation and B-Notices land on clients.

### What was explored but not pursued
- **SFTP/Zapier/Slack integrations**: Settings page shows them as LIVE but they're not wired. Deferred — not on BookMule's core feature list and not blocking accountant V1.
- **AP/Bill Pay**: BookMule lists this as a feature but it's substantial (approval workflows, ACH/check/card payment initiation, PO-bill matching). Marked P2 — likely V2 given complexity.
- **Invoice templates**: BookMule has these, but not in Will's V1 priorities for the accountant ICP. Deferred to P2.

### Current state
The codebase is more complete than expected on the core QBO write operations — import, edit, delete, and export are all genuinely wired with real Inngest functions and QBO API calls. The gaps are: (1) token refresh (silent failure after 1 hour), (2) receipt scan OCR (infrastructure exists, just needs the Claude vision call), (3) revert operations (UI promise with no backend), (4) safe merge (completely absent), and (5) the full 1099 filing flow beyond creating TaxJob records. The strategic gap is the vendor canonicalization engine and Plaid integration that differentiate the product from filing-only tools.

- Then receipt scan OCR — the upload API and Inngest skeleton exist, just needs the Claude vision API call wired in
