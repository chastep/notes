# Sterling → BookMule Gap Analysis & Remaining Build Plan

## Summary

This session audited the full Sterling codebase against BookMule's feature set to identify what's wired end-to-end versus what's still missing. Import (CSV → QBO), mass-edit, mass-delete, and export are all fully wired with real Inngest background jobs and QBO API calls — significantly more complete than expected. The strategic context from Will's notes (target accountants not direct SMB; vendor canonicalization is the moat; Plaid bank feeds are V1 priority; IRS applications need to be filed now) was incorporated into the gap analysis. A second pass added Stripe real-credential wiring and all three plan enforcement gates (row quotas, entity count, seats) — currently none of the limits are enforced even though the `PlanTier` enum exists on `Workspace`.

## Key Topics

- Full codebase audit: pages, API routes, Inngest functions, Prisma models
- BookMule feature set: 8 core capabilities including safe merge, invoice templates, AP/Bill Pay, 1099 Center
- Will's strategic notes: accountant ICP, per-vendor pricing, vendor canonicalization as moat, Plaid as V1 must-have
- QBO token refresh gap — blocks all operations after ~1 hour of inactivity
- Stripe: checkout/portal/webhook are wired but price IDs are placeholders — real credentials still needed
- Plan enforcement: row quotas, entity count limits, and seat limits are not enforced anywhere today
- 1099 filing flow: W-9 chase emails, TIN validation, 1099-NEC PDF generation, IRS FIRE export
- Vendor canonicalization: fuzzy match + embeddings + LLM reasoning to unify vendor name variants across sources

## Decisions & Insights

- Import, mass-edit, mass-delete, and export are production-ready — the core QBO write operations are done
- Receipt scan and 1099 Center are "UI + data model only" — no real AI processing or filing wired
- Safe merge (consolidate vendors/customers/accounts) is completely absent from the codebase
- QBO token refresh is the single most urgent fix — every operation will fail after ~1 hour without it
- "Reverts are free" is a marketing claim with no backend implementation yet
- Stripe checkout/portal/webhook exist but price IDs are `placeholder` strings — nothing real transacts
- `Workspace` has no `monthlyRowsUsed` or `billingPeriodStart` fields — schema migration required before quota enforcement
- FREE tier row limit (100/mo), entity limit (1), and seat limit (1) are unenforced today
- The strategic differentiation vs Track1099/Tax1099 is vendor canonicalization + bank-feed cross-reference, not the filing itself
- IRS TIN Matching e-services and FIRE/IRIS TCC applications need to be filed immediately (4-8 week approval lag)

## Code Snippets

### QBO Token Refresh — src/lib/providers/qbo.ts
```typescript
export async function ensureFreshToken(entity: Entity): Promise<Entity> {
  if (entity.tokenExpiresAt && entity.tokenExpiresAt > new Date()) return entity;
  const res = await fetch('https://oauth.platform.intuit.com/oauth2/v1/tokens/bearer', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded',
      Authorization: `Basic ${Buffer.from(`${process.env.QBO_CLIENT_ID}:${process.env.QBO_CLIENT_SECRET}`).toString('base64')}`,
    },
    body: new URLSearchParams({ grant_type: 'refresh_token', refresh_token: entity.refreshToken }),
  });
  const data = await res.json() as { access_token: string; refresh_token: string; expires_in: number };
  return db.entity.update({
    where: { id: entity.id },
    data: {
      accessToken: data.access_token,
      refreshToken: data.refresh_token,
      tokenExpiresAt: new Date(Date.now() + data.expires_in * 1000),
    },
  });
}
```

### Row Quota Enforcement — src/lib/pricing.ts
```typescript
const ROW_LIMITS: Record<string, number | null> = {
  FREE: 100,
  STARTER: null,
  MULTI_ENTITY: null,
};

export async function checkRowQuota(workspace: Workspace, rowCount: number) {
  const limit = ROW_LIMITS[workspace.plan];
  if (limit === null) return;
  const periodAge = Date.now() - workspace.billingPeriodStart.getTime();
  if (periodAge > 30 * 24 * 60 * 60 * 1000) {
    await db.workspace.update({ where: { id: workspace.id }, data: { monthlyRowsUsed: 0, billingPeriodStart: new Date() } });
    return;
  }
  if (workspace.monthlyRowsUsed + rowCount > limit) {
    throw new QuotaError(`Plan limit: ${workspace.monthlyRowsUsed}/${limit} rows used this month`);
  }
}
```

## Claude Handoff

### What we worked on
This session covered two arcs. First: a full product gap analysis comparing Sterling's codebase against BookMule's feature set, incorporating Will's strategic notes about targeting accountants over direct SMB and the vendor canonicalization engine being the product moat. Second: a focused pass on Stripe wiring and plan enforcement — identifying that while the Stripe checkout/portal/webhook routes exist, they use placeholder price IDs and none of the plan limits (rows, entities, seats) are actually enforced in the API routes. Both sets of findings were written into the plan file at `/Users/chasestephens/.claude/plans/resilient-orbiting-scroll.md`.

### Key decisions and why
- **Token refresh is P0**: Every QBO operation silently fails after ~1 hour. It's a one-file fix with maximum leverage.
- **Stripe + enforcement are P0 together**: There's no point enforcing limits until the upgrade flow actually transacts. They need to ship as a unit — real price IDs → test checkout → webhook updates plan → enforcement gates kick in.
- **Schema migration before enforcement**: `monthlyRowsUsed` and `billingPeriodStart` don't exist on `Workspace` yet. Enforcement code can't be written until the migration runs.
- **Vendor canonicalization is the moat** (per Will): The differentiation vs filing-only tools is resolving name variants across QBO + bank feeds. Plaid is a prerequisite, not an add-on.
- **IRS applications need to be filed this week** (per Will): TIN Matching and FIRE/IRIS TCC have 4-8 week approval lags. Format-only TIN validation is the V1 fallback while waiting.

### What was explored but not pursued
- **SFTP/Zapier/Slack integrations**: Settings page shows them as connected but they're not wired. Deferred — not blocking accountant V1.
- **AP/Bill Pay**: Substantial build (approval workflows, payment initiation, PO-bill matching). Marked P2/V2.
- **Invoice templates**: Not in Will's V1 accountant priorities. Deferred to P2.

### Current state
Core QBO write operations (import, edit, delete, export) are genuinely production-ready. The three big gaps for going live are: (1) QBO token refresh — silent failure after ~1 hour, (2) Stripe real credentials + enforcement gates — no limits are enforced today, and (3) the 1099 filing flow beyond TaxJob record creation. The vendor canonicalization engine and Plaid integration are the V1 differentiators that still need to be built from scratch.
