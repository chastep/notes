# Sterling — From AI Agency Concept to Accountant SaaS

## Summary

This project started as a Y Combinator-inspired idea: build an AI agency that replaces professional services headcount with AI agents, starting with 1099 contractor tax processing. Through domain review from Will Rosler (tax expert) and iterative design, the model evolved: the core product is now **Sterling** — a self-serve SaaS tool for accountants targeting solo CPAs and small bookkeeping firms with 50+ SMB clients. The 1099 Center is a $4/form AI-managed add-on rather than a standalone managed service. The core insight from the original concept survived: charge for vendor cleanup and canonicalization (where the real work is), bundle the filing. The codebase has shipped Plans 1–3 (QBO import/edit/delete/export, Stripe billing scaffolding, Clerk auth, history/audit). Four plans remain.

## The Original Insight (still true)

A tax domain expert confirmed: the real complexity in 1099 processing isn't the filing — it's getting the correct vendor list first. QBO exports aren't cash-basis-aware, CC payments must be excluded, C-corps don't qualify, and the $2,000 threshold (2026, per OBBBA) applies to cash payments only. DIY filing tools (Track1099, Tax1099) expect clean data — Sterling produces the clean data. That's the moat.

**Vendor canonicalization is the product:** Resolving "Joe's Plumbing LLC" (QBO) + "JOSEPHS PLUM ACH 7/15" (Chase bank) + "Joe Smith Plumbing" (prior W-9 on file) = same vendor. Fuzzy match + embedding similarity + LLM reasoning + confidence scoring. High confidence = auto-accept. Medium = generate accountant review question. Low = flag for manual resolution.

## Business Model

**Core:** Self-serve SaaS for accountants. Connect client QBO companies, run bulk operations (import, edit, delete, export, receipt scan).

**Pricing tiers:**
- FREE — 100 rows/month, 1 entity, 1 user
- STARTER — $25/mo, unlimited rows, 1 entity, 3 users
- MULTI_ENTITY — $79/mo, unlimited rows, 5 entities, 10 users

**1099 Center add-on:** $4/form, AI-managed — Sterling does vendor canonicalization, W-9 chasing, and generates the IRS filing. Accountant reviews and approves.

**Distribution:** Accountants (solo CPAs, small bookkeeping firms, 2–15 people with 50+ SMB clients). One accountant = 20–200 SMB clients. Volume leverage, B2B sales motion, longer relationships, lower CAC than chasing direct SMB.

**Competitive wedge:** Filing-only tools don't do the upstream cleanup. Accounting firms do but charge $1,500–5,000+/season. Sterling charges for cleanup + bundles filing. Don't compete on filing price — charge for the work that saves accountant hours.

## Four Remaining Plans

### Plan 4 — Production Hardening *(next sprint)*

Makes what's already built reliable and billable.

- **QBO token refresh** — `ensureFreshToken(entity)` helper in `src/lib/providers/qbo.ts`; call Intuit refresh endpoint when `tokenExpiresAt` is past; thread through all three Inngest functions
- **Stripe real credentials** — create Starter ($25/mo) + Multi-entity ($79/mo) products in Stripe dashboard; swap placeholder price IDs in env; test checkout → webhook → `workspace.plan` update end-to-end
- **Schema migration** — add `monthlyRowsUsed Int @default(0)` and `billingPeriodStart DateTime @default(now())` to `Workspace`
- **Row quota enforcement** — `checkRowQuota` helper in `src/lib/pricing.ts`; FREE = 100 rows/mo; return 402 + `upgradeRequired: true` in all 4 submit routes
- **Entity count enforcement** — in QBO connect callback, check entity count vs plan limit (FREE/STARTER = 1, MULTI_ENTITY = 5); redirect to upgrade if at limit
- **`<UpgradePrompt />`** — shared inline component shown when any gate returns 402
- **Revert operations** — `src/app/api/operations/[id]/revert/route.ts` + `src/inngest/functions/revert-operation.ts`

### Plan 5 — 1099 Managed Service *($4/form AI-managed add-on)*

Ships the primary V1 revenue mechanism.

- **Vendor canonicalization engine** — `src/lib/vendor-canonicalization.ts`: fuzzy match + embedding similarity + LLM review; `VendorAlias` model on `Entity`; confidence scoring (≥90% auto-accept, 70–89% accountant question, <70% flag)
- **W-9 chase** — `src/app/api/1099/chase-w9/route.ts` via Resend; magic-link no-login intake at `src/app/(public)/w9/[token]/page.tsx`
- **TIN format validation** — `src/lib/tin-validation.ts` (format only in V1; IRS TIN Matching API V2 after approval clears)
- **1099-NEC PDF generation** — `pdf-lib` per-vendor PDFs; IRS-spec layout
- **IRS FIRE flat file** — IRS Publication 1220 format; accountant downloads + uploads manually in V1; IRIS API in V2 (mandatory by Dec 31, 2026)
- **Threshold logic** — $2,000 for 2026 (OBBBA); multi-year back-filing (2024/2025/2026)
- **Stripe per-form billing** — `price_1099_form` one-time charge on `TaxJob` completion

**Non-code prerequisites (file this week):** IRS TIN Matching e-services application (irs.gov) + IRS FIRE Transmitter Control Code (fire.irs.gov) + IRIS registration. All have 4–8 week approval lags.

### Plan 6 — Accountant Platform

Builds the multi-client workspace that makes Sterling an accountant's daily tool.

- **Multi-client dashboard** — `Client` Prisma model; `src/app/(dashboard)/dashboard/clients/page.tsx` with client list, job status, flag queue, W-9 status per client
- **Plaid bank feed integration** — `src/lib/plaid.ts`; Link flow; nightly sync → `BankTransaction` model; required for canonicalization to cross-reference cash payments against QBO vendor names
- **Prior-year 1099 + W-9 upload** — skip re-chasing known vendors; parse prior W-9 TINs and cross-reference against current year list
- **Receipt scan OCR** — wire Claude claude-sonnet-4-6 vision API into existing Inngest skeleton (`src/inngest/functions/scan-receipt.ts`)
- **Safe merge** — `src/app/(dashboard)/dashboard/merge/page.tsx` + `src/inngest/functions/merge-qbo.ts`; consolidate duplicate vendors/customers/accounts with pre-merge snapshot for revert

### Plan 7 — Scale & Polish

Fills remaining BookMule feature gaps and hardens for growth.

- Export XLSX + email delivery (Resend) + scheduled exports (Inngest cron)
- Cross-entity operations (fan out import/export/edit/delete across all workspace entities)
- Member invites (Clerk org invitations) + seat enforcement in invite route
- IRS IRIS API migration (replaces FIRE — mandatory Dec 31, 2026)
- Live IRS TIN Matching API (once approval clears)
- QBO App Store listing (V2 — requires Intuit certification, ~6–8 weeks)

## Key Decisions

- **Accountants, not direct SMB**: One accountant = 20–200 SMB clients. Cleaner ICP, B2B sales motion, lower CAC, repeat revenue. Bill.com made this exact bet.
- **Per-vendor pricing for 1099**: $8–12/vendor, $200–300 minimum (to A/B-test against flat tier). Aligns price with the cleanup work, not the filing. Easy for accountants to mark up to clients.
- **Plaid is V1, not V2**: Without bank feed data, vendor canonicalization only has one source (QBO). The product becomes "a glorified Track1099." Plaid cross-reference is what makes the canonical vendor graph meaningful.
- **QBO App Store is V2**: Discoverability is real but puts the product on Intuit's roadmap radar, constrains to QBO-only, and has mediocre conversion. Better as a supplementary channel after accountant validation.
- **IRS e-filing is manual V1**: FIRE flat file download + manual upload. IRS IRIS API replaces this by Dec 31, 2026 (mandatory transition). Register for IRIS now even though V1 uses FIRE.

The design spec is at `docs/superpowers/specs/2026-05-26-sterling-roadmap-design.md`.

### Key decisions and why
- **Accountant ICP over direct SMB**: Volume leverage — one accountant = 20-200 clients. Cleaner targeting on LinkedIn (bookkeeper, fractional CFO, CAS lead). Better CAC, longer relationships.
- **Vendor canonicalization is the moat**: The differentiation isn't filing speed — it's resolving name variants across QBO + bank feeds. Track1099/Tax1099 expect clean data; Sterling produces it. This is the IP that compounds over time as the vendor graph grows.
- **Plaid is Plan 6, not optional**: Without bank data the canonicalization engine only has QBO names. One data source = limited differentiation. Two sources = the "we already have this vendor's payment history" graph that no filing tool can replicate.
- **Production hardening before new features (Plan 4 first)**: Token refresh is a silent failure mode. Stripe has placeholder price IDs. Zero plan limits enforced. Shipping features on top of this foundation creates customer trust issues when they hit the silent failures. Fix the foundation first.
- **$2,000 threshold for 2026**: OBBBA passed and reverted the previously planned lower thresholds. The $600 threshold referenced in earlier docs is wrong for tax year 2026. Multi-year support needed because accountants back-file for clients who skipped.
